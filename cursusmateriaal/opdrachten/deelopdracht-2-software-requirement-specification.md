# Deelopdracht 2: Nieuwe user stories bedenken en verwerken in SRS

## Deelopdracht 2:  
**Nieuwe user stories bedenken en verwerken in SRS**

##  **Doel**  

Je leert hoe je nieuwe wensen van een klant vertaalt naar user stories en deze verwerkt in een bestaande Software Requirement Specification (SRS). Daarnaast leer je hoe je een backlog opstelt en een definition of done maakt.

##  **Beschrijving**  

De klant is erg blij met het voorlopige ontwerp, maar heeft twee nieuwe wensen doorgegeven die nog niet in het ontwerp stonden.

  1. **Donkere modus** : De klant wil graag dat bezoekers kunnen schakelen naar een "dark mode", zodat de website ook in de avond prettig leesbaar is.
  2. **Social media delen** : Studenten willen hun projecten direct kunnen delen op LinkedIn en Twitter via een knop op de projectpagina.

Aan jou de taak om deze nieuwe wensen te vertalen naar user stories en ze toe te voegen aan het bestaande SRS document. Daarna maak je een backlog voor het hele project.

**Stappen**

  * **Stap 1: Lees het bestaande SRS document**

    * Pak het SRS document erbij.
    * Lees de bestaande functionele eisen door om te zien hoe ze zijn opgeschreven.
  * **Stap 2: Vertaal de nieuwe wensen naar user stories**

    * Maak een user story voor de "Donkere modus". Formuleer deze met: 
      * **Wie** : (bijv. bezoeker)
      * **Wat** : (bijv. thema wisselen)
      * **Waarom** : (bijv. prettiger lezen in het donker)
    * Maak een user story voor "Social media delen". Formuleer deze ook volgens het wie-wat-waarom format.
    * Zorg dat beide user stories realistisch en duidelijk zijn.
  * **Stap 3: Verwerk de user stories in de SRS**

    * Voeg de twee nieuwe user stories toe aan het SRS document.
    * Geef ze een logisch nummer (bijv. FE10.1 en FE10.2 of een nieuwe categorie).
    * Zorg dat ze passen in de stijl van het document.
  * **Stap 4: Maak een definition of done**

    * Bedenk wanneer een user story klaar is.
    * Schrijf op wat er moet gebeuren voordat een user story "done" is (bijv. getest, documentatie, code review).
    * Maak een lijst met criteria voor "done".
  * **Stap 5: Maak een backlog**

    * Maak een overzicht (backlog) met daarin de nieuwe user stories Ã©n minimaal 5 andere belangrijke user stories uit het SRS.
    * Zet bij elke user story: 
      * Nummer
      * Wie (gebruiker)
      * Wat (functionaliteit)
      * Waarom (reden)
    * Voeg de definition of done toe aan dit document.
  * **Stap 6: Stem af met stakeholders**

    * Controleer of de nieuwe user stories duidelijk zijn voor de ontwikkelaars.
    * Vraag feedback aan je "opdrachtgever" (docent/begeleider) over de formulering.

 

## **Op te leveren**

  * Een aangepast SRS document (of een kopie ervan) waarin de twee nieuwe user stories zijn verwerkt.
  * Een backlog document met daarin de nieuwe user stories, een aantal bestaande stories en de definition of done.

 

##  **Beoordeling/toetsing**  

De beoordeling vindt plaats middels voltooid/niet voltooid.

 

##  **Beoordelingsinstrument**  

Wanneer heb je de opdracht behaald?

  * Zijn de twee nieuwe wensen correct vertaald naar user stories?
  * Zijn de user stories correct toegevoegd aan het SRS document?
  * Is de backlog en definition of done compleet en duidelijk?
